# Индивидуальный отчет по "Проектный практикум 1"

## Моя роль

В рамках проекта "Учебная платформа" я выступала в роли backend-разработчика, отвечающего за модуль тестов с автоматической проверкой. Моя зона ответственности включала работу с моделью данных тестовых вопросов, бизнес-логикой проверки ответов и API для взаимодействия с тестами.

## Перечень выполненных задач

В ходе работы над проектом мной были выполнены следующие задачи:
- Проектирование модели данных  
  Создана модель `TestQuestion`, включающая текст вопроса, варианты ответов, правильный ответ, порядковый номер вопроса, связь с уроком. Учтено мягкое удаление (архивация) и каскадное удаление при удалении урока.
- Разработка Pydantic-схем валидации данных
- Реализация слоя доступа к данным `TestQuestionRepository`
- Разработка сервисного слоя `TestQuestionService` с бизнес-логикой
- Разработка API для модуля тестирования  
  Реализованы эндпойнты:
  - CRUD-операции над тестовыми вопросами (для администратора)
  - получение теста к уроку без правильных ответов (для студентов)
  - проверка ответов пользователя с обратной связью
  - получение оценки за урок

## Использованные технологии и инструменты

- **Python**
- **FastAPI**
- **PostgreSQL** — база данных проекта
- **SQLAlchemy** — описание моделей, асинхронная работа с БД, построение запросов
- **Alembic** — управление миграциями базы данных
- **Pydantic** — валидация входных и выходных данных, строгие схемы API
- **Docker** — контейнеризация приложения и базы данных
- **uv** — управление зависимостями и запуск проекта в изолированной среде
- **uvicorn** – ASGI-сервер для запуска приложения
- **Git** – система контроля версий

## Итоги

Реализован модуль тестирования, интегрированный с общей архитектурой платформы. Получен опыт проектирования сервисной архитектуры на стеке Python/FastAPI, освоены новые технологии и инструменты. В целом участие в учебном проекте дало понимание основных принципов реальной разработки.

**Основные трудности**
- Необходимость тщательного проектирования на ранних этапах для минимизации правок в процессе разработки (потребовалось изменить типы полей модели и сделать миграции)
- Необходимость правильно разделять ответственность между слоями архитектуры

**Впечатления от командной работы**
Положительный опыт:
- Четкое разделение ответственности благодаря предварительной договоренности
